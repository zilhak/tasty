# 프로젝트 전략 관점

라이브러리 분리가 tasty의 장기 비전, AI 에이전트 생태계 기여, 경쟁 차별화에 미치는 영향을 분석한다.

---

## 프로젝트 장기 비전과의 정합성

CLAUDE.md에 명시된 tasty의 핵심 정체성:

> cmux(macOS 전용)에서 영감을 받은 크로스 플랫폼 GPU 가속 터미널 에뮬레이터.
> AI 코딩 에이전트에 특화된 기능을 제공한다.

핵심 키워드: **크로스 플랫폼**, **GPU 가속**, **AI 에이전트 특화**.

### 분리와 비전의 정합성

| 후보 | 비전 정합성 | 이유 |
|------|-----------|------|
| `tasty-hooks` | **높음** | AI 에이전트 자동화의 핵심 빌딩 블록. 분리하면 에이전트 생태계 전체에 기여. |
| `tasty-terminal` | **높음** | 헤드리스 터미널 엔진은 AI 에이전트가 GUI 없이 셸을 제어하는 핵심 인프라. |
| `tasty-renderer` | **중간** | GPU 렌더러 공유는 "GPU 가속 터미널" 생태계 확장에 기여하지만, 분리 난이도가 높음. |
| `tasty-model` | **중간** | 데이터 모델의 추상화는 백엔드 교체 가능성을 열지만, 현재 우선순위 아님. |
| `tasty-ipc-*` | **낮음** | JSON-RPC 프로토콜은 범용이지만 tasty 고유 가치가 아님. |
| `tasty-notification` | **낮음** | 알림은 UI 기능이지, 에이전트 자동화의 핵심이 아님. |
| `tasty-settings` | **낮음** | 설정은 완전히 내부 관심사. |

## AI 에이전트 생태계 기여 가능성

### tasty-hooks: 에이전트 자동화의 표준 인터페이스

현재 AI 코딩 에이전트(Claude Code, Cursor, Aider, Continue 등)는 터미널을 "맹목적으로" 사용한다. 명령을 보내고 출력을 읽지만, **이벤트 기반 반응**은 불가능하다.

`tasty-hooks`가 독립 크레이트로 공개되면:

1. **에이전트 프레임워크가 직접 통합**: 에이전트가 "빌드 완료 시 다음 단계 실행", "에러 패턴 감지 시 자동 수정" 같은 이벤트 기반 워크플로우를 구축할 수 있다.

2. **다른 터미널 에뮬레이터 채택**: WezTerm, Alacritty 등이 같은 훅 시스템을 도입하면, 에이전트 도구가 터미널 에뮬레이터에 구애받지 않고 동작.

3. **표준화 가능성**: 터미널 이벤트 훅의 de facto 표준이 될 수 있음.

### tasty-terminal: 에이전트의 셸 인터페이스

AI 에이전트가 셸 명령을 실행하고 결과를 읽는 데 사용하는 현재 방식:

```
에이전트 → std::process::Command → 출력 캡처
```

이 방식의 한계:
- 대화형(interactive) 프로그램 제어 불가
- 실시간 출력 스트리밍 불가
- 프롬프트 감지 불가
- VTE 시퀀스 처리 불가

`tasty-terminal`이 제공하는 것:

```
에이전트 → tasty-terminal::Terminal → PTY + VTE 파싱 + 이벤트 + Read Mark
```

- 대화형 프로그램 완전 지원 (PTY)
- 실시간 출력 (process() + events)
- 출력 델타 트래킹 (set_mark/read_since_mark)
- VTE 시퀀스 정상 처리 (termwiz 파서)

이것은 에이전트가 "진짜 터미널"과 동일한 방식으로 셸을 제어할 수 있게 한다.

## 타 터미널 프로젝트와의 차별화

### 경쟁 터미널 프로젝트의 라이브러리 전략

| 프로젝트 | 라이브러리 분리 | 외부 재사용 |
|----------|---------------|------------|
| **Alacritty** | `alacritty_terminal` 크레이트 분리 | crates.io 공개, 일부 프로젝트 사용 |
| **WezTerm** | `termwiz` 크레이트 분리 | crates.io 공개, tasty도 사용 중 |
| **Rio** | 단일 크레이트 | 분리 없음 |
| **Kitty** | C 코드, 라이브러리 분리 없음 | N/A |
| **Zed (terminal)** | 내부 workspace 크레이트 | 외부 공개 없음 |

### tasty의 차별점

Alacritty는 "터미널 엔진"을, WezTerm은 "VTE 파서"를 분리했다. 둘 다 **저수준 인프라**에 집중.

tasty가 **AI 에이전트 특화 라이브러리**를 분리하면 완전히 다른 포지셔닝:

- `tasty-hooks`: 이벤트 기반 자동화 (최초)
- `tasty-terminal`: Read Mark API 포함 에이전트용 터미널 (최초)

이것은 tasty를 "또 다른 터미널 에뮬레이터"가 아닌 **"AI 에이전트 인프라 프로젝트"**로 포지셔닝한다.

## 분리 타이밍: 지금 vs 나중

### 지금 분리하는 것이 유리한 후보

**tasty-hooks**와 **tasty-terminal**:

- 커플링이 이미 0 (hooks) 또는 최소 (terminal).
- 코드가 성장할수록 커플링이 증가할 위험. 지금이 가장 깨끗한 시점.
- 외부 의존을 형성하면 API 안정화 압력이 코드 품질을 높임.
- AI 에이전트 생태계가 급성장 중. 선점 가치.

### 나중에 분리해야 하는 후보

**tasty-model**, **tasty-renderer**:

- 현재 API가 안정되지 않음. model.rs는 계속 구조 변경 중.
- trait 추상화 설계가 필요한데, 요구사항이 아직 불확실.
- 성급한 추상화는 나중에 되돌리기 어려움.

### 분리하지 않아야 하는 후보

**tasty-ipc-protocol**, **tasty-ipc-server**, **tasty-notification**, **tasty-settings**:

- 외부 재사용 가치 없음.
- 분리하면 관리 부담만 증가.
- 영원히 tasty 내부에 있어도 문제 없는 코드.

## 분리하지 않을 때의 리스크

### 단기 (현재 ~ 1만 줄)

리스크 없음. 8,870줄은 단일 크레이트로 관리하기에 적절한 규모.

### 중기 (1만 ~ 2만 줄)

- **빌드 시간 증가**: 증분 빌드 시간이 30초 이상으로 늘어나면 개발 생산성 저하.
- **테스트 격리 부재**: GUI 의존 없이 실행 가능한 테스트를 분리하지 못하면, CI 환경 구성이 복잡해짐.
- **기여 장벽**: 전체 프로젝트를 이해해야 작은 수정도 할 수 있는 구조는 오픈소스 기여를 저해.

### 장기 (2만 줄 이상)

- **아키텍처 부채**: 모듈 간 경계가 불명확해지면 순환 의존 발생 위험.
- **리팩토링 비용 폭증**: 나중에 분리하려면 이미 형성된 커플링을 끊어야 하므로 비용이 기하급수적으로 증가.
- **생태계 기회 상실**: AI 에이전트 생태계의 성장기를 놓치면 다른 프로젝트가 같은 영역을 선점.
