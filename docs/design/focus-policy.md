# 포커스 정책

> **상태: 구현 완료.** 멀티 윈도우 포커스 정책 적용됨. 모달 포커스 차단 구현됨.

## 계층 구조

```
Engine
└── Modal (최대 1개) ─── 있으면 모든 입력을 독점
└── Window (여러 개) ─── Modal이 없을 때 OS 네이티브 포커스
    └── Pane/Surface ── 윈도우 내부 포커스
```

## 규칙

### 1. Modal이 없을 때

각 윈도우는 OS 네이티브 포커스 정책을 따른다.

- Window 1과 Window 2는 독립적으로 포커스를 받을 수 있다
- 윈도우 사이에 다른 앱의 창이 있을 수 있다 (z-order 독립)
- 윈도우 내부의 Pane/Surface 포커스는 해당 윈도우가 관리

### 2. Modal이 있을 때

Modal이 열리면, 엔진이 `modal_active` 플래그를 설정한다.

- **모든 윈도우**는 키보드/마우스 입력을 무시한다
- Modal 윈도우만 입력을 받는다
- Modal을 닫아야만 윈도우에 다시 포커스가 간다

### 3. Modal의 종류

| 종류 | 예시 | 동작 |
|------|------|------|
| Modal | 설정창 | 전체 입력 차단, 닫기 전까지 다른 조작 불가 |
| Window | 알림창, 터미널 윈도우 | 독립 포커스, 다른 윈도우와 공존 |

### 4. 포커스 차단 구현

OS 네이티브 윈도우 비활성화(Win32 `EnableWindow` 등)를 사용하지 않는다. 플랫폼마다 동작이 달라서 크로스플랫폼 일관성이 깨진다.

대신 앱 레벨에서 처리:

- 엔진이 `modal_active: AtomicBool`을 보유
- 각 윈도우의 이벤트 핸들러가 매 이벤트마다 확인
- `modal_active == true`이면 키보드/마우스 이벤트를 무시
- 윈도우가 시각적으로 비활성 상태임을 표시 (반투명 오버레이 등)

### 5. 윈도우 내부 포커스

Modal/Window 레벨 포커스와 별개로, 각 윈도우 내부에서:

- 패인 간 포커스: Alt+Arrow 또는 클릭
- 서피스(SurfaceGroup) 간 포커스: Alt+Arrow 또는 클릭
- 탭 전환: Ctrl+Tab

이 내부 포커스는 해당 윈도우가 OS 포커스를 가지고 있고, Modal이 활성화되지 않았을 때만 동작한다.

### 6. CLI/IPC 포커스 독립 원칙

**focus는 사용자의 시선과 관심을 나타내는 독립적 행위이며, CLI/IPC 명령의 대상을 결정하는 수단이 아니다.**

focus를 명령의 대상 결정에 사용하면 다음과 같은 문제가 발생한다:

- 명령 발행 후 실행 전에 사용자가 focus를 이동하면 의도하지 않은 대상에 명령이 실행된다
- AI 에이전트가 명령을 보내놓고 다른 작업을 위해 focus를 옮기면 원래 의도가 깨진다
- focus 상태에 의존하는 명령은 본질적으로 race condition을 내포한다

따라서:

- 모든 명령에 `--surface`, `--pane` 등 명시적 대상 지정 파라미터가 있어야 한다
- 호출자가 대상을 알아야 할 때는, 먼저 현재 상태를 조회(`list surfaces` 등)하여 ID를 확인한 뒤 해당 ID를 명령에 전달한다
- 리소스를 생성하는 명령은 응답에 생성된 리소스의 ID를 반드시 포함해야 한다
- `set focus` 계열 명령은 예외 — focus 자체를 조작하는 것이 목적
