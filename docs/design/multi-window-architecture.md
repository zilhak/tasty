# 멀티 윈도우 아키텍처

> **상태: 설계 완료, 미구현.** 현재는 단일 윈도우(`App` 구조체) 구조. 이 문서는 전환할 목표 아키텍처를 기술한다.

## 구조

```
Engine (1개 프로세스, 메인 스레드)
├── IPC Server (TCP JSON-RPC, 단일 포트)
├── Window 1 (독립 OS 윈도우, 자체 wgpu 서피스 + egui 컨텍스트)
├── Window 2 (독립 OS 윈도우, 자체 wgpu 서피스 + egui 컨텍스트)
├── Window N ...
└── Modal (최대 1개, 전역)
```

## 엔진

프로세스의 메인 루프. 다음을 소유한다:

- **IPC 서버**: 단일 포트. CLI(`tasty` 명령)와 외부 프로그램은 엔진에 요청하고, 엔진이 해당 윈도우에 전달.
- **전역 상태**: 워크스페이스, 알림, 훅, 설정 등 윈도우 간 공유 상태.
- **윈도우 생명주기**: 윈도우 생성/파괴.
- **모달 상태**: `modal_active` 플래그.

## 윈도우

각 윈도우는 독립된 OS 윈도우이며, 하나의 프로세스 내에서 스레드로 동작한다.

- 자체 wgpu 서피스 + egui 컨텍스트 보유
- wgpu adapter/device는 엔진에서 공유
- 윈도우 간 통신은 엔진을 통해 수행 (직접 통신 없음)
- 각 윈도우는 여러 워크스페이스/패인/탭을 가질 수 있음

## 모달

전역적으로 최대 1개만 존재. 설정창이 대표적.

- 모달이 열리면 엔진이 `modal_active` 플래그를 설정
- 모든 윈도우는 매 이벤트마다 이 플래그를 확인하여, 활성 시 키보드/마우스 입력을 무시
- 모달이 닫히면 플래그 해제, 윈도우들이 다시 입력을 받음
- 모달 자체도 독립 OS 윈도우 (자체 wgpu 서피스 + egui)

## 프로세스/스레드 선택 근거

멀티 프로세스가 아닌 멀티 스레드를 채택한다.

| 관점 | 근거 |
|------|------|
| 상태 공유 | Arc/Mutex로 엔진 상태 직접 공유. IPC 불필요 |
| GPU 리소스 | wgpu adapter/device를 윈도우 간 공유 가능 |
| winit 호환 | winit은 프로세스당 하나의 이벤트 루프를 전제로 설계됨 |
| 크래시 격리 | PTY 프로세스(쉘)는 이미 별도 OS 프로세스. 쉘 크래시가 Tasty에 전파되지 않음 |
| 리소스 폭주 방어 | 스크롤백 버퍼 상한 + PTY 읽기 채널 버퍼 제한으로 방어 |

Chrome이 멀티 프로세스인 이유(신뢰할 수 없는 웹 코드의 보안 격리)는 Tasty에 해당하지 않는다.
