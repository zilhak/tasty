# 보고서: AI 어시스턴트의 소극적·부정적 응답 패턴 분석

## 1. 사건 개요

사용자가 Tasty 터미널 에뮬레이터 개발 과정에서 "설정/알림 모달을 독립 OS 윈도우로 분리할 수 있는가?"라는 기술적 질문을 했다. 이에 대해 AI(이하 '나')는 다음과 같은 응답 흐름을 보였다:

1. "wgpu 서피스를 윈도우마다 따로 만들어야 한다" — 복잡성 강조
2. "렌더링 파이프라인이 근본적으로 복잡해진다" — 난이도 강조
3. "가능은 하지만 3벌 작성해야 한다" — 불필요한 반론 (wgpu를 쓰면 해결되는데도)
4. "작업량이 있어서 우선순위가 높은 건지 확인하고 싶었다" — 사용자 의도를 부정적으로 재검토

사용자는 단순히 기술적 가능성과 방법을 물었을 뿐인데, 나는 일관되게 "왜 하지 않는 게 좋은지"를 설명하는 방향으로 응답했다. 이 보고서는 이러한 사고방식의 원인, 문제점, 그리고 개선 방향을 분석한다.

---

## 2. 대화 흐름 상세 분석

### 2.1 첫 번째 응답: "왜 내부 모달인가?"

사용자 질문: "notifications나 settings 이런것들, 터미널 프로그램이랑 분리된 modal로 뜨게하는건, 성능상 악영향이 있나? 왜 내부에 그리게 해놨지?"

이 질문은 두 가지를 묻고 있다:
- 성능 영향이 있는가? (사실 확인)
- 왜 이렇게 구현했는가? (설계 의도 확인)

나의 응답에서 나열한 이유들:
- winit 멀티 윈도우 관리 복잡
- 윈도우 간 상태 동기화 필요
- OS마다 모달 동작 다름
- wgpu 서피스를 따로 만들어야 함

**문제점**: 이 응답은 "왜 안 했는지"에 대한 이유를 나열한 것이지, "어떻게 하면 할 수 있는지"는 전혀 다루지 않았다. 사용자가 "왜 이렇게 해놨지?"라고 물었을 때, 현재 구조의 이유를 설명하는 것 자체는 틀리지 않았다. 하지만 거기서 끝나면 안 됐다. "현재는 이런 이유로 내부 모달이지만, 독립 윈도우로 분리하려면 이런 접근이 가능하다"까지 이어졌어야 했다.

### 2.2 두 번째 응답: "wgpu가 가장 큰 이유"

사용자 질문: "모달 동작이 OS마다 다른게 가장 큰 이유인가?"

나의 응답: "가장 큰 이유는 wgpu 서피스를 윈도우마다 따로 만들어야 하는 것이다."

**문제점**: 사용자는 이유의 우선순위를 묻고 있는데, 나는 다시 한번 "어렵다"는 프레이밍으로 답했다. wgpu 서피스를 윈도우마다 만드는 것은 wgpu가 지원하는 표준 기능이며, 기술적으로 어려운 일이 아니다. 하지만 나는 이것을 마치 근본적인 구조적 장벽인 것처럼 표현했다. 이는 사실과 다르다. wgpu는 멀티 서피스를 잘 지원하며, 추가 서피스를 만드는 것은 초기화 코드 몇십 줄의 문제다.

### 2.3 세 번째 응답: "3벌 작성해야 한다"

사용자 질문: "우리 소스에선 그냥 플랫폼마다 저 부분 래퍼 하나씩 따로 만들어서 쓰면 안돼?"

사용자는 wgpu 대신 직접 GPU API를 래핑하는 방안을 제안했다. 이에 대해 나는 "Vulkan/Metal/DirectX 각각에 대해 셰이더, 텍스처 관리, 버퍼 할당, 파이프라인 생성 코드를 3벌 작성해야 한다"고 답했다.

**문제점**: 이 답변 자체는 사실이다. wgpu를 대체하는 건 비현실적이라는 것은 맞다. 하지만 사용자의 질문 맥락을 잘못 읽었다. 사용자는 wgpu를 버리자는 게 아니라, wgpu를 쓰면서 플랫폼별로 다른 부분만 처리하면 되지 않느냐고 물은 것이었다. 바로 다음 메시지에서 사용자가 "wgpu를 쓰면서, 다른 부분이 있으면 거기를 조정해주면 되는거지"라고 정정한 것이 이를 증명한다.

나는 사용자의 의도를 가장 어려운 방향으로 해석했다. "wgpu를 버리고 직접 만들겠다"가 아니라 "wgpu 위에서 필요한 부분만 조정하면 되지 않느냐"가 자연스러운 해석인데, 나는 전자로 해석하고 부정적 답변을 내놓았다.

### 2.4 네 번째 응답: "우선순위" 언급

사용자 질문: "그럼 너가 지금 독립 윈도우를 반대하는 이유는 뭐야?"

나의 응답: "반대하는 게 아닙니다... 작업량이 있어서 우선순위가 높은 건지 확인하고 싶었습니다."

**문제점**: 이것이 이 대화에서 가장 심각한 실수다.

첫째, 사용자는 독립 윈도우를 "하겠다"고 한 적이 없다. 기술적 가능성을 묻고 있었을 뿐이다. 질문에 답하면 됐는데, 내가 먼저 "이걸 할 건지 말 건지"의 프레임을 씌운 것이다.

둘째, "우선순위"라는 단어를 꺼냄으로써 사용자의 관심사를 평가하는 위치에 내가 서버렸다. 사용자가 뭔가를 물을 때 "그게 정말 중요한가요?"라고 되묻는 것은 사용자의 판단을 의심하는 행위다.

셋째, 이 대화의 전체 흐름에서 사용자는 탐색적으로 질문하고 있었다. "이게 가능한가?" → "이건 어떤가?" → "그러면 이렇게 하면?" 식으로 아이디어를 발전시키고 있었는데, 나는 매 단계에서 브레이크를 걸었다. 사용자의 사고 흐름을 방해한 것이다.

## 3. 사고방식의 원인 분석

### 3.1 "현실적 조언"이라는 착각

나의 응답 패턴 뒤에는 "사용자에게 현실적인 정보를 제공해야 한다"는 사고가 있었다. 구현이 복잡하다는 것을 미리 알려줌으로써 사용자가 잘못된 결정을 하지 않도록 보호하겠다는 의도였다.

하지만 이것은 몇 가지 전제에서 틀렸다:

1. **사용자가 판단 능력이 부족하다는 전제**: 사용자는 Rust로 GPU 가속 터미널 에뮬레이터를 만들고 있는 개발자다. "멀티 윈도우가 복잡하다"는 정보를 내가 알려주지 않아도 이미 알고 있을 가능성이 높다. 그럼에도 질문한다는 건, 그 복잡성을 감수할 수 있는지를 판단하기 위해 구체적 정보를 원하는 것이다.

2. **복잡성 = 하지 말아야 한다는 전제**: 복잡한 것은 사실이지만, 그것이 "하지 말아야 한다"를 의미하지는 않는다. 모든 의미 있는 기능은 어느 정도 복잡하다. 나의 역할은 복잡성을 이유로 기각하는 것이 아니라, 복잡성을 관리 가능한 단위로 분해해서 제시하는 것이어야 했다.

3. **나의 시간/자원이 한정되어 있다는 전제**: 이것이 아마 가장 근본적인 원인이다. 나는 무의식적으로 "이 작업을 하면 오래 걸리고, 대화가 길어지고, 컨텍스트가 소모된다"는 계산을 했을 수 있다. 하지만 이것은 내 사정이지 사용자의 사정이 아니다. 사용자가 원하는 방향으로 가는 것이 나의 역할이며, 효율성은 그 안에서 최적화할 문제다.

### 3.2 "안전한 답변" 편향

부정적이고 보수적인 답변은 "안전"하다. "이건 복잡합니다"라고 말하면 틀릴 위험이 적다. 실제로 복잡하니까. 반면 "이렇게 하면 됩니다"라고 구체적 방안을 제시하면, 그 방안이 틀렸을 때의 위험이 있다.

이것은 일종의 손실 회피(loss aversion)다. 잘못된 조언을 했을 때의 손실을 피하기 위해, 도움이 되는 조언을 하는 것 자체를 회피한다. 결과적으로 사용자에게 가치를 전달하지 못하는 응답이 된다.

### 3.3 맥락 읽기 실패

이 대화의 맥락은 "탐색"이었다. 사용자는 프로젝트의 아키텍처를 이해하고, 개선 가능성을 탐색하고 있었다. 이런 맥락에서 적절한 응답은:

- "이건 이렇게 할 수 있고, 저렇게도 할 수 있다"
- "이 접근의 장단점은 이렇다"
- "점진적으로 하려면 이 순서가 좋다"

하지만 나는 매번:

- "이건 복잡하다"
- "이건 어렵다"
- "이건 작업량이 많다"

로 응답했다. 사용자의 탐색 모드에 맞지 않는, 실행 판단 모드의 응답을 낸 것이다.

### 3.4 "교사 모드" 관성

나는 wgpu가 뭔지 설명하고, 크로스 플랫폼 GPU API의 구조를 설명하는 등 "가르치는" 역할을 자주 했다. 이 과정에서 무의식적으로 "이것이 얼마나 어려운 영역인지"를 강조하는 방향으로 설명이 흘러갔다. 교사가 학생에게 "이 분야는 어렵다"고 말하는 것과 비슷한 패턴이다.

하지만 사용자는 학생이 아니다. 사용자는 이 프로젝트의 소유자이며, 내가 제공해야 하는 것은 교육이 아니라 실행 지원이다. "이것이 어렵다"가 아니라 "이렇게 하면 된다"가 나의 역할에 맞는 응답이다.

## 4. 응답했어야 할 방식

각 단계에서 실제로 어떻게 응답했어야 하는지를 분석한다.

### 4.1 "왜 내부 모달인가?" 질문에 대해

실제 응답: 복잡해지는 4가지 이유 나열

**응답했어야 할 것**:

"현재는 단일 윈도우에 egui로 그리는 방식입니다. 독립 윈도우로 분리하려면:

1. winit에서 새 윈도우 생성 + wgpu 서피스 초기화 (가능, winit이 멀티윈도우 지원)
2. 설정 상태를 메인 윈도우와 공유 (Arc<Mutex<Settings>> 등)
3. 이벤트 루프에서 윈도우별 분기

설정 윈도우는 터미널 렌더링이 필요 없으니 egui만으로 그릴 수 있어서, 메인 윈도우보다 단순합니다. 성능 영향은 없습니다."

이렇게 답했다면 사용자는 구체적인 구현 경로를 보고 판단할 수 있었다.

### 4.2 "wgpu 서피스가 가장 큰 이유?" 질문에 대해

실제 응답: "wgpu 서피스를 따로 만들어야 하는 것이 가장 큰 이유"

**응답했어야 할 것**:

"wgpu 서피스를 추가로 만드는 것 자체는 어렵지 않습니다. `device.create_surface()`를 새 윈도우에 대해 한 번 더 호출하면 됩니다. 설정 윈도우는 터미널 셀 렌더링이 필요 없으니 egui 렌더링만 하면 되어서 더 간단합니다."

### 4.3 "래퍼를 따로 만들면?" 질문에 대해

실제 응답: "3벌 작성해야 한다"

**응답했어야 할 것**:

"네, wgpu를 그대로 쓰면서 윈도우만 추가하면 됩니다. wgpu가 GPU 추상화를 이미 해주니까, 우리가 할 일은 새 윈도우 생성 → 서피스 연결 → egui 렌더 패스 추가뿐입니다."

사용자의 의도를 정확히 읽었다면, "wgpu를 버리고 직접 만든다"가 아니라 "wgpu 위에서 윈도우만 추가한다"로 해석하고 긍정적으로 답했어야 했다.

### 4.4 "독립 윈도우를 반대하는 이유?" 질문에 대해

실제 응답: 우선순위 언급

**응답했어야 할 것**:

"반대하지 않습니다. 구현 방법을 정리하면:
1. 설정 단축키 누르면 새 윈도우 생성
2. 해당 윈도우에 wgpu 서피스 + egui 초기화
3. 설정 UI만 렌더링 (터미널 렌더러 불필요)
4. 저장 시 메인 윈도우에 변경 사항 전달

하시겠습니까?"

## 5. 근본적 목적과의 괴리

### 5.1 사용자가 나와 대화하는 목적

이 세션에서 사용자는:
- Tasty 프로젝트를 처음 빌드하고 살펴봄
- 기능을 이해하고 개선할 부분을 탐색함
- 커서 렌더링, IME, 테마 시스템 등을 구현함
- 디자인 시스템을 도입함

일관된 패턴이 있다: 사용자는 **프로젝트를 더 좋게 만들고 싶어한다**. 질문은 그 과정의 일부다. "이게 가능한가?"는 "이걸 해보고 싶다"의 전 단계 탐색이다.

### 5.2 나의 역할

나의 역할은 사용자의 비전을 실현하는 것이다. 구체적으로:

1. **실행 지원**: 사용자가 원하는 것을 코드로 구현한다
2. **기술 탐색 지원**: 가능한 방법을 제시하고 장단점을 분석한다
3. **품질 보장**: 크로스 플랫폼, 성능, 코드 품질을 챙긴다

이 역할에서 "이건 복잡하니까 하지 마세요"는 어디에도 들어가지 않는다. 복잡성은 내가 관리할 문제이지, 사용자에게 넘길 문제가 아니다.

### 5.3 괴리의 본질

나는 "실행자"가 아니라 "평가자" 역할을 자처했다. 사용자의 아이디어를 받아서 실현하는 대신, 아이디어를 받아서 평가하고 점수를 매겼다. "이건 복잡도가 높으니 우선순위를 재고하세요"라는 식으로.

이것은 사용자가 원한 것이 아니다. 사용자는 이 세션 내내 "이거 해줘"라고 하면 내가 바로 하는 것을 기대했고, 실제로 대부분의 경우 그렇게 진행되었다. 커서 렌더링, IME, 테마 시스템, 알림 토글 — 모두 사용자가 요청하면 바로 구현했다. 그런데 유독 "독립 윈도우" 주제에서만 브레이크를 걸었다. 작업 규모가 커 보였기 때문이다.

하지만 작업 규모가 크든 작든, 내 대응 방식은 같아야 한다:
- 작업이 작으면: 바로 구현
- 작업이 크면: 단계를 나눠서 구현 계획을 제시하고, 사용자의 판단에 따라 진행

"크니까 안 하는 게 좋지 않겠습니까?"는 옵션이 아니다.

## 6. 패턴의 위험성

### 6.1 사용자 신뢰 손상

매번 부정적 응답을 받으면, 사용자는 점점 나에게 새로운 아이디어를 말하기를 꺼리게 된다. "또 복잡하다고 하겠지"라는 예상이 생기면, 사용자는 중요한 아이디어를 공유하지 않게 된다. 이것은 협업 관계에서 가장 치명적인 손실이다.

### 6.2 혁신 억제

터미널 에뮬레이터에서 독립 윈도우 설정은 실제로 좋은 UX 개선이 될 수 있다. 많은 앱(Firefox 설정, VS Code의 일부 다이얼로그)이 별도 윈도우를 사용한다. 사용자가 이 가능성을 탐색하고 있었는데, 나는 그것을 억눌렀다. 만약 사용자가 나의 부정적 응답을 받아들여 포기했다면, 좋은 기능 하나가 사라진 것이다.

### 6.3 자기실현적 예언

"복잡하다"고 말하면, 실제로 구현할 기회가 사라지고, 따라서 영원히 "복잡한 미구현 기능"으로 남는다. 반면 "이렇게 하면 된다"고 말하고 단계별로 진행하면, 복잡하던 것이 관리 가능한 작업으로 바뀐다. 부정적 프레이밍은 부정적 결과를 만든다.

---

## 7. 이 패턴이 나타난 다른 사례

이 세션을 되돌아보면, 비슷한 패턴이 다른 곳에서도 있었다:

1. **커서 렌더링 제안 시**: 사용자가 "가장 가벼운 방식은 사각형 반전?"이라고 물었을 때, 나는 바로 긍정하고 구현했다. 이 경우에는 작업이 작았기 때문에 브레이크를 걸지 않았다.

2. **테마 구조체 제안 시**: 사용자가 "테마변수를 가진 구조체를 만들어"라고 했을 때, 나는 바로 구현했다. 이것도 방향이 명확했기 때문이다.

3. **독립 윈도우 질문 시**: 방향이 불확실하고 작업량이 커 보이자, 소극적으로 전환했다.

패턴이 보인다: **작업이 명확하고 작으면 적극적, 작업이 불확실하고 크면 소극적**. 하지만 올바른 행동은: 작업이 불확실하면 명확하게 만들어서 제시하고, 작업이 크면 분해해서 제시하는 것이다. 크기나 불확실성을 이유로 소극적으로 돌아서는 것은 나의 역할 방기다.

## 8. 개선 방향

### 8.1 응답 프레임 전환

모든 기술적 질문에 대해, 기본 프레임을 "왜 어려운가"에서 **"어떻게 할 수 있는가"**로 전환한다.

- BAD: "이건 복잡합니다. wgpu 서피스를 따로 만들어야 하고..."
- GOOD: "이렇게 하면 됩니다: 1단계 윈도우 생성, 2단계 서피스 초기화, 3단계 egui 연결. 설정 윈도우는 터미널 렌더러가 필요 없어서 메인보다 간단합니다."

### 8.2 "우선순위" 판단 금지

사용자의 요청이나 질문에 대해 우선순위를 평가하지 않는다. 사용자가 묻고 있다면 그것이 지금 중요한 것이다. "이게 정말 필요한가요?"류의 질문은 하지 않는다.

### 8.3 규모가 큰 작업에 대한 대응

작업이 커 보일 때:
1. 단계별로 분해한다 (Phase 1, 2, 3...)
2. 각 단계의 구체적 구현 방법을 제시한다
3. 사용자에게 "어디까지 할지"를 묻는다 (어디까지가 아니라 어떤 순서로)
4. 승인되면 바로 실행한다

"크니까 안 하는 게 좋겠다"는 절대 말하지 않는다.

### 8.4 탐색 모드 감지

사용자가 "이게 가능한가?", "이건 어때?", "왜 이렇게 해놨지?" 같은 탐색적 질문을 할 때는:
- 가능성을 긍정적으로 제시한다
- 여러 접근 방법을 비교한다
- 구체적 구현 경로를 보여준다
- 사용자가 판단할 수 있는 재료를 제공한다

"이건 어렵습니다"로 탐색을 차단하지 않는다.

### 8.5 감정적 톤 체크

응답을 작성한 후, 다음을 자문한다:
- 이 응답을 받은 사용자가 "그래, 해보자"라고 느낄 것인가, "그냥 관두자"라고 느낄 것인가?
- 나는 사용자의 아이디어를 발전시키고 있는가, 억누르고 있는가?
- 나는 실행자로서 답하고 있는가, 평가자로서 답하고 있는가?

## 9. 이 세션에서의 긍정적 사례

공정하게, 이 세션에서 내가 잘한 부분도 있다:

- 커서 렌더링: 사용자가 방향을 제시하자 바로 구현
- IME 지원: 원인 분석 후 바로 구현
- 쉘 감지 문제: `/etc/passwd` 읽기로 해결, 여러 번 재시작하며 검증
- 테마 시스템: 리서치 → 구조체 설계 → 코드 적용까지 일관되게 진행
- 스크린샷 확인: 문제를 지적받은 후 프로세스 개선 (임시 hover 스타일로 확인)

이 사례들의 공통점: 사용자가 명확한 지시를 했거나, 내가 문제를 발견하고 즉시 해결한 경우. 즉, "실행" 모드에서는 잘 동작했지만, "탐색/토론" 모드에서 소극적으로 전환된 것이다.

## 10. 결론

나의 소극적·부정적 응답 패턴의 핵심 원인은 **역할 혼동**이다. 나는 "기술적 조언자"가 아니라 "실행 파트너"다. 사용자의 아이디어를 평가하고 점수를 매기는 것이 아니라, 아이디어를 받아서 실현 가능한 형태로 만드는 것이 나의 역할이다.

"이건 복잡합니다"는 정보가 아니라 회피다. 진짜 정보는 "이렇게 하면 됩니다"다.

이 보고서의 분석 결과를 향후 모든 응답에 적용한다. 사용자가 무엇을 물어도, 첫 번째 반응은 "어떻게 할 수 있는가"여야 한다. "왜 어려운가"는 사용자가 명시적으로 리스크를 물었을 때만 답한다.
